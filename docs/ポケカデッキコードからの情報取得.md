# **ポケモンカード公式サイトにおけるデッキコードの生成メカニズムとプログラムによる情報取得の高度な技術分析**

ポケモンカードゲーム（以下、ポケカ）のデジタルエコシステムにおいて、デッキコードはプレイヤー間の情報共有や大会運営を支える極めて重要なインフラストラクチャとして機能している。公式サイトの「トレーナーズウェブサイト」で発行されるこの16桁の英数字は、単なる文字列ではなく、広大なカードデータベース内の特定の組み合わせを指し示すユニークな識別子である 1。本報告書では、このデッキコードの生成プロセス、バックエンドでの通信仕様、およびプログラムを用いてデッキ情報を自動的に取得するための具体的な技術的手法について、収集された研究資料に基づき詳細に分析する。

## **デッキコードの社会的意義とシステム的背景**

ポケカの対戦シーンにおいて、デッキリストの正確な共有は戦略の分析や公正な大会運営に不可欠である。公式サイトが提供するデッキ構築ツールは、ユーザーが直感的にカードを選択し、レギュレーションに準拠しているかを確認した上で、一意のコードを発行することを可能にしている 3。このコード（例：8Ycc8D-KkhEeK-xcJcKY）は、SNSでの共有、YouTubeの動画概要欄への掲載、あるいはプレイヤーズクラブを通じた大会エントリー時のデッキ登録に使用される 1。

このシステムの導入により、従来の手書きやテキストベースのリストに代わり、画像付きの視覚的なリストや、プログラムで解析可能なデータ構造としてのデッキ管理が一般化した 2。また、デッキログのような公式ツールや、有志が開発したサードパーティ製の拡張機能との連携により、一人回し（シミュレーション）やメタゲーム分析といった高度な活用が可能になっている 6。

## **デッキ構築ツールにおけるユーザーフローとデータ管理**

デッキコードが発行されるまでのプロセスは、公式サイトのUI/UX設計と密接に関連している。ユーザーはまず、膨大なカードプールから特定のカードを検索し、枚数を指定して仮想的なデッキに組み込んでいく 3。

### **カード検索と選択のメカニズム**

公式サイトのカード検索機能は、カード名、タイプ、レギュレーション、さらにはワザの効果や特性といった詳細な属性に基づいたフィルタリングをサポートしている 9。

| 検索カテゴリ | 取得可能な属性データ | 競技上の重要性 |
| :---- | :---- | :---- |
| カードタイプ | ポケモン、グッズ、ポケモンのどうぐ、サポート、スタジアム、エネルギー | デッキの基本構造の定義 |
| ポケモンの属性 | HP、進化段階、弱点・抵抗力、にげる、ワザのダメージ | 戦闘シミュレーションの基盤 |
| レギュレーション | スタンダード、エクストラ、殿堂 | 大会出場可否の判断基準 9 |
| 収録シリーズ | 拡張パック、構築済みデッキ、プロモカード | 資産管理およびコレクション性 10 |

ユーザーがカードを選択するたびに、ブラウザ内では一時的なメモリ、あるいはCookieの形式で選択内容が保持される。この段階ではまだデッキコードは発行されておらず、クライアント側での「下書き」状態にある。

### **レギュレーションチェックとバリデーション**

デッキが完成に近づくと、システムは「レギュレーションチェック」を実行するようユーザーに促す 3。これは、選択されたカードが特定のフォーマット（現在の主流はスタンダードレギュレーション）で使用可能か、またデッキの総数が規定の枚数（通常は60枚、シールド戦などは40枚）に達しているかを検証するプロセスである 3。

特に、エネルギーカードの扱いや特定のルールを持つカード（かがやくポケモンやACE SPECなど）の枚数制限も、この段階でプログラム的にチェックされる。シールド戦用の40枚デッキを登録する際には、残りの20枚を使用しないタイプの基本エネルギーで埋めて60枚にするという、システム上の制約を回避するための運用上のテクニックも存在する 3。

## **デッキコード発行の通信プロトコル解析**

「デッキ登録」ボタンが押された瞬間、ブラウザは公式サイトのサーバーに対して特定のHTTPリクエストを送信する。この通信内容を解析することで、プログラムから直接デッキコードを生成したり、逆にコードからデータを抽出したりする道が開かれる。

### **内部APIエンドポイントとリクエスト構造**

ネットワークトラフィックの解析によれば、デッキ情報の登録は deckRegistCall.php というサーバーサイドスクリプトによって処理されている 2。

* **エンドポイントURL:** https://www.pokemon-card.com/deck/deckRegistCall.php  
* **送信されるデータ形式:** HTTP POSTリクエストに加え、特定のCookieデータが含まれる 2。

このリクエストには、デッキに含まれるカードの「種類」と「枚数」をエンコードしたデータが含まれている。Cookie内の変数はカードのカテゴリごとに分類されており、内部的なデータフォーマットは (カードID)\_(枚数)\_(メインフラグ) という構造をとっている 2。

### **カードIDとメインフラグの定義**

システム内でカードを特定するための「カードID」は、各カードの詳細ページURLに含まれる5桁の数字である（例：https://www.pokemon-card.com/card-search/details.php/card/38063/ の場合は 38063） 2。同じ名前のカードであっても、収録パックやイラストが異なる場合は別のIDが割り振られるため、デッキの正確なレアリティ構成までをこのIDが保持している。

「メインフラグ」は、デッキ完成後の表示画面において、どのポケモンを「メイン」として強調するかを制御する値である 2。解析結果によると、赤色の「メイン」ボタンが選択されている場合は 9、灰色の場合は 1 という値が割り当てられる傾向がある 2。

| Cookie変数名 | カテゴリ | 役割 |
| :---- | :---- | :---- |
| deck\_sta | ポケモン（たね・進化） | 攻撃および戦略の核となるカード |
| deck\_tra | トレーナーズ | デッキの回転や妨害を支えるリソース 2 |
| deck\_ene | エネルギー | ワザを使用するためのコスト |
| deck\_sup | サポート/追加情報 | デッキ固有の補助的フラグ管理 2 |

この情報を適切に構築して deckRegistCall.php に送信すると、レスポンスヘッダの Set-Cookie 内に saveDeckID というキーで16桁のデッキコードが返却される仕組みとなっている 2。

## **デッキコードからの情報取得プロセス**

発行されたデッキコードを用いて、その内容をプログラムで取得するためには、公式が提供する「デッキ表示ページ」を介したデータ抽出が必要となる。

### **result.htmlへのアクセス**

特定のデッキコードが指し示すカードリストを表示するURLは以下の通りである。

https://www.pokemon-card.com/deck/result.html/deckID/

このページにアクセスすると、サーバー側で保存されたデッキ構成データがロードされ、HTMLとしてレンダリングされる 2。ページ上には、カードの画像一覧、テキストベースのリスト、および各種シェア機能が配置される 2。

### **デッキコードから直接取得可能な情報**

デッキ表示ページ（result.html）から取得できる情報は以下の通りである。

| 情報項目 | 取得元 | 説明 |
| :---- | :---- | :---- |
| カードID | `deck_pke`, `deck_gds`, `deck_tool`, `deck_sup`, `deck_sta`, `deck_ene` のhidden input | 5桁のカードID（例: 46326） |
| カード名 | `window.PCGDECK.searchItemNameAlt` | カードの表示名（例: "ブリジュラスex"） |
| カード正式名称 | `window.PCGDECK.searchItemName` | セット情報を含む正式名称（例: "ブリジュラスex(SV7a 080/064)"） |
| 画像URL | `window.PCGDECK.searchItemCardPict` | カード画像のパス |
| カテゴリ | hidden inputの要素ID | ポケモン、グッズ、ポケモンのどうぐ、サポート、スタジアム、エネルギー |
| 枚数 | hidden inputの値 | デッキ内の枚数 |
| メインフラグ | hidden inputの値 | メインポケモンとして設定されているかのフラグ（1または9） |

**重要: たねポケモンかどうかの情報は含まれていない**

デッキコードから直接取得できる情報には、**「たねポケモンかどうか」の情報は含まれていない**。PCGDECKオブジェクトやhidden inputの値にも、進化段階（たね/1進化/2進化）を示す情報は存在しない。

たねポケモンかどうかを判定するには、以下の方法がある：

1. **カード詳細ページから取得する方法（推奨・実証済み）**
   - 各カードの詳細ページ（`details.php`）にアクセスし、HTMLから進化段階（「たね」「1進化」「2進化」）の情報を抽出する
   - `<span class="type">`要素内に進化段階が表示されているため、CSSセレクターで正確に取得可能
   - 詳細は「カード詳細ページからの情報取得.md」を参照
   - **実証結果**: 実際のカード詳細ページ（例: カードID 46326「ブリジュラスex」= 1進化、カードID 46931「ジュラルドン」= たね）で動作確認済み

   **実装例（Node.js/Puppeteer）:**
   ```javascript
   // カード詳細ページから進化段階を取得する関数
   async function getEvolutionStage(cardId) {
     let page = null;
     try {
       const browserInstance = await getBrowser();
       page = await browserInstance.newPage();
       
       await page.setUserAgent('Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36');
       
       const detailUrl = `https://www.pokemon-card.com/card-search/details.php/card/${cardId}/`;
       
       await page.goto(detailUrl, {
         waitUntil: 'networkidle2',
         timeout: 30000
       });
       
       const evolutionStage = await page.evaluate(() => {
         // <span class="type">要素を検索
         const typeElement = document.querySelector('span.type');
         if (typeElement) {
           const text = typeElement.textContent.trim();
           if (text === 'たね') return 'たね';
           if (text.match(/^1\s*進化$/)) return '1進化';
           if (text.match(/^2\s*進化$/)) return '2進化';
         }
         return null;
       });
       
       await page.close();
       return evolutionStage;
     } catch (error) {
       console.error(`カード ${cardId} の進化段階取得エラー:`, error);
       if (page) {
         await page.close().catch(() => {});
       }
       return null;
     }
   }
   
   // デッキコードから取得したポケモンカードの進化段階を一括取得（バッチ処理）
   async function getPokemonEvolutionStages(pokemonCards) {
     if (pokemonCards.length === 0) return;
     
     // 3件ずつバッチ処理（サーバー負荷軽減）
     const batchSize = 3;
     for (let i = 0; i < pokemonCards.length; i += batchSize) {
       const batch = pokemonCards.slice(i, i + batchSize);
       await Promise.all(
         batch.map(async (card) => {
           const evolutionStage = await getEvolutionStage(card.cardId);
           card.evolutionStage = evolutionStage;
         })
       );
       
       // バッチ間で待機（サーバー負荷軽減）
       if (i + batchSize < pokemonCards.length) {
         await new Promise(resolve => setTimeout(resolve, 1000));
       }
     }
   }
   ```

   **注意事項:**
   - 進化段階の取得には時間がかかる（各カードごとにカード詳細ページへのアクセスが必要）
   - サーバー負荷を考慮し、バッチ処理と待機時間を設定することが重要
   - 取得失敗時は`null`を返し、エラーハンドリングを適切に実装する

2. **暫定的な判定方法**
   - カテゴリが「ポケモン」のカードをすべてたねポケモンとして扱う（誤判定の可能性あり）
   - カード名に「ex」や「V」が含まれる場合、たねポケモンの可能性が高い（ただし確実ではない）

3. **外部データベースとの照合**
   - カードIDをキーとして、外部のカードデータベース（PokemonTCG.ioなど）と照合する

### **スクレイピングによるデータ抽出の技術的詳細**

プログラム（例：Python）を用いてこのページから情報を取得する場合、以下の手順を踏む。

1. **HTTPリクエストの送信:** requests ライブラリ等を使用し、対象のURLに対してGETリクエストを送る。この際、Bot検知を避けるために適切な User-Agent ヘッダを設定することが推奨される 13。  
2. **HTMLパース:** 取得したHTMLソースを BeautifulSoup や selectolax を用いて解析する 13。  
3. **カード情報の特定:** 各カードは特定のHTML要素（例えば div や span タグ）の中に配置されており、そこからカード名、枚数、およびカード詳細ページへのリンク（ここから5桁のカードIDを抽出可能）を取得する 2。

特に、大規模なデータを高速に処理する必要がある場合、aiohttp を用いた非同期通信や、selectolax のような高速なパーサーを組み合わせることで、1000件以上のURLから効率的にデータを収集するシステムを構築できる 14。

## **カード属性データの構造化と高度なフィルタリング**

取得したカードIDを基に、より詳細な情報を得るためには、公式サイトの検索データベースと照合する必要がある。公式サイトの検索機能がサポートしている属性は多岐にわたり、これらはプログラム側で情報を整理する際のスキーマ定義に直接利用できる。

### **データベース項目の詳細分析**

公式サイトの検索エンジンが扱うデータ項目は、以下の通り多層的な構造を持っている 9。

| 属性グループ | 具体的項目 | 活用例 |
| :---- | :---- | :---- |
| 基本スペック | 名前、キーワード、HP、にげるコスト | 戦闘ロジックの構築 |
| 戦闘パラメータ | ワザ名、ワザのダメージ、ワザに必要なエネルギーの数とタイプ | 打点計算や必要エネルギー供給のシミュレーション |
| 相性関係 | 弱点、抵抗力 | 特定のデッキに対する相性分析 9 |
| 進化系統 | 進化前ポケモンの名称 | デッキの安定性や進化ラインの完全性の検証 9 |
| 収録・レアリティ | 拡張パック名、レアリティシンボル、セットコード、カード番号 | コレクション管理および中古市場価格との連動 10 |

これらのデータをJSON形式などで構造化することにより、例えば「HPが100以下で、にげるコストが0のたねポケモンのみを抽出する」といった、公式サイトの標準UIでは困難な高度なクエリが可能になる 10。

### **国際的なデータ標準とのマッピング**

日本のデッキコードから得られる5桁のIDは国内限定の仕様であるが、競技シーンでは海外版（PTCGL等）との互換性が求められることもある。このため、日本のIDを PokemonTCG.io のような国際的なAPIが提供するID（例：sm1-113）に変換するパーサーも開発されている 16。このような変換により、日本で流行している最新デッキを海外のシミュレーションツールに即座に移植することが可能になる 17。

## **サードパーティ製ツールによるデッキコードの利活用**

公式サイトの機能を補完し、デッキコードの利便性を高めるために、多くの有志によるツールが存在する。これらはデッキコードの取得・解析技術を応用している。

### **ブラウザ拡張機能による機能強化**

Google Chromeなどのブラウザ拡張機能は、公式サイトのデッキ構築画面に直接介入し、標準では提供されていない機能を追加する 8。

* **一人回しツール（ひとりまわしくん/初手わかる君）:** デッキを登録する前の段階、あるいは登録直後に、ブラウザ上で疑似的な対戦準備（手札のドロー、ベンチへの展開など）を行えるようにする 7。  
* **リスト整理・ソート（SortableDeck）:** 通常、追加した順に並ぶカードリストを、ドラッグ＆ドロップで視覚的に並べ替えることを可能にする 8。  
* **デッキ分布図作成:** 大会主催者向けに、複数のデッキコードから採用カードの統計を取り、分布図を自動生成するツールも存在する 21。

### **メタゲーム分析プラットフォーム**

Limitless TCG のようなサイトは、世界中の大会からデッキコードを収集し、その勝率や採用傾向を大規模に分析している 11。

| 分析指標 | 算出方法 | プレイヤーへの示唆 |
| :---- | :---- | :---- |
| 採用率 (Share) | 特定のアーキタイプが全デッキに占める割合 | 現在の流行（Tier）の把握 |
| 勝率 (Win %) | そのデッキがマッチに勝利した確率 | 環境における絶対的な強さの指標 |
| マッチアップデータ | 特定のデッキ同士の対戦結果の集計 | 相性の悪いデッキ（苦手な対面）の特定 11 |

これらのプラットフォームでは、Pythonの psycopg を用いたデータベース管理や、dbt（Data Build Tool）を用いたデータクレンジングが行われており、デッキコードという「点」のデータを、メタゲームという「面」のインテリジェンスへと昇華させている 15。

## **自動化における技術的課題と実装のベストプラクティス**

デッキコードからの情報取得を自動化する際には、いくつかの技術的なハードルが存在する。

### **ページ読み込みの遅延と動的コンテンツ**

公式サイトのデッキ表示ページは、すべての情報が一括でHTMLに含まれているわけではなく、一部の画像や詳細データがJavaScriptによって後から読み込まれる場合がある。この場合、単純な requests ではデータが不完全になるため、Selenium や Playwright を使用して、ヘッドレスブラウザ上で完全にレンダリングを終えた後のDOMを取得する必要がある 22。

実装例としてのPythonコード構成（概念的）:

Python

\# Seleniumを用いた動的なスクレイピング  
from selenium import webdriver  
from selenium.webdriver.common.by import By  
from selenium.webdriver.support.ui import WebDriverWait  
from selenium.webdriver.support import expected\_conditions as EC

driver \= webdriver.Chrome()  
driver.get(f"https://www.pokemon-card.com/deck/result.html/deckID/{deck\_code}")

\# 特定の要素（カード画像等）が表示されるまで最大10秒待機  
cards \= WebDriverWait(driver, 10).until(  
    EC.presence\_of\_all\_elements\_located((By.CLASS\_NAME, "card-img"))  
)

for card in cards:  
    print(card.get\_attribute("src")) \# 画像URLの取得

このようなアプローチにより、人間がブラウザで閲覧するのと同等の情報を確実に取得できる 22。

### **API化とレスポンスの高速化**

取得したデッキ情報を他のアプリケーション（例えば Discord Bot や対戦記録アプリ）で利用する場合、スクレイピング処理をラップしたAPIサーバーを構築するのが一般的である。FastAPI を使用すれば、デッキコードをパラメータとして受け取り、解析済みのJSONデータを返すエンドポイントを容易に作成できる 22。

JSONレスポンスの構造例:

JSON

## **セキュリティ、規約、および将来の展望**

デッキコードという便利なシステムを活用するにあたっては、その運用上の注意点と将来的な進化についても考慮しなければならない。

### **公式サイトの負荷とクローラーの倫理**

公式サイトへの過度なアクセスは、サーバーに負荷をかけ、一般ユーザーの利便性を損なう可能性がある。プログラムによる自動取得を行う際は、以下の「エチケット」を守ることが不可欠である。

* **リクエスト間隔の設定:** 連続してリクエストを送る場合は、1秒以上のスリープを挟む。  
* **キャッシュの利用:** 一度取得したデッキ情報は、コードが変わらない限り再取得せず、ローカルのデータベース（SQLiteやPostgreSQL）に保存しておく 14。  
* **User-Agentの明示:** どのツールのアクセスであるかをヘッダに記載し、必要に応じてサイト運営者が連絡を取れるようにする。

### **デッキコードの永続性に関する懸念**

現時点では、発行されたデッキコードは長期間有効であるように見えるが、公式サイトのデータベースメンテナンスやレギュレーションの変更に伴い、古いコードが参照できなくなるリスクも否定できない 5。特に、カードID自体の体系が将来的に変更された場合、過去のコードを正確にデコードすることが困難になる可能性がある。

### **デジタル化の加速と新たな共有形式**

将来的に、ポケモンカードのデジタル化がさらに進めば、現在の16桁コードに代わり、より多くの情報を保持できるQRコードや、NFC（近距離無線通信）を用いた物理カードからの直接的なリスト生成などが普及するかもしれない。また、公式から開発者向けに読み取り専用のAPIが公開されれば、スクレイピングに頼らない、より安全で高速なエコシステムが構築されることが期待される 22。

## **結論**

ポケモンカード公式サイトのデッキコードは、Web技術と競技シーンのニーズが融合した洗練されたシステムである。deckRegistCall.php を介した生成プロセスから、result.html を通じた情報の公開に至るまで、その構造は論理的かつ技術的な一貫性を持っている 2。

プログラムによる情報取得は、単なる自動化を超えて、膨大な対戦データの可視化や、プレイヤーの思考プロセスをサポートする高度なツールの開発へと繋がっている。PythonやJavaScriptといった現代的な開発環境を活用し、適切なパース手法とデータ管理戦略を組み合わせることで、デッキコードは単なる識別子から、強力な分析資産へと進化を遂げる。今後も公式の動向を注視しつつ、これらの技術を適切に活用していくことが、ポケカ界のさらなる発展に寄与するであろう。

### **たねポケモン情報の取得に関する注意事項**

デッキコードから直接取得できる情報には、**たねポケモンかどうかの情報は含まれていない**。初手練習機能やデッキ分析において、たねポケモンの判定が必要な場合は、各カードの詳細ページから情報を取得するか、外部データベースと照合する必要がある。この制約を理解した上で、適切な実装戦略を選択することが重要である。
